# 02 — UX Plan: The Lesson Primitive and the Shape of Learning

> Slice owner: `ux`. Sibling docs: `01-product.md` (strategy), `03-architecture.md` (architecture), `04-brand.md` (positioning). The `Step` schema in §1 is shared with `arch` and is the load-bearing contract for everything else.

---

## 0. The pivot

Today: one lesson = one full Python exercise (10–40 lines, 5–15 min staring at a blank editor). For an AI-first builder coming from PM/ops/marketing, that step is too tall — the blank-page tax is the dropout tax. The new atom is the **Step**: one screen, 30–90 seconds, one commitment. A chapter becomes a sequenced ladder of 12–20 Steps. Reading is interleaved with doing. The user produces working code by minute four — but only because the surface they touched at minute one was a single radio button.

---

## 1. The Step primitive (the load-bearing section)

### 1.1 Core type — shared with `arch`

```ts
type Step =
  | ReadStep | MultipleChoiceStep | FillBlankStep | PredictOutputStep
  | FixBugStep | WriteCodeStep | CodeReorderStep | CheckpointQuizStep;

interface BaseStep {
  id: string;                   // ULID, stable across edits
  kind: Step["kind"];
  chapterSlug: string;
  lessonSlug: string;           // a lesson groups 12–20 steps
  index: number;                // 0-based position within lesson
  phase: "warmup" | "build" | "check";
  concept: string;              // e.g. "for-loop-iteration"
  estSeconds: number;           // 30–90, used for daily-goal math
  xp: number;                   // 5/8/12/20 by difficulty
  prompt: string;               // markdown, ≤ 240 chars
  hints?: string[];             // progressive reveal, max 3
}

type StepAttempt = {
  stepId: string; startedAt: string; submittedAt: string;
  correct: boolean; payload: unknown; hintsUsed: number;
};
```

All step results normalize to `StepAttempt` so the runtime never branches on type.

> **Schema reconciliation with `arch` (03-architecture.md §1):** the architecture doc is canonical for runtime types and authoring. Type-name mapping: `multiple-choice → mc`, `fill-blank → fill`, `predict-output → predict`, `fix-bug → fix`, `write-code → write`, `code-reorder → reorder`, `checkpoint-quiz → checkpoint`. `arch` also provides typed `Grader` (`string-equality` / `stdout-equality` / `ast-match` / `llm-judge`) and a richer `Hint = { level: 1|2|3, body, cost? }`, replacing my flat `hints: string[]`. Personalization tokens (`{{user.name}}`, `{{user.dog}}`, `{{user.team}}`, `{{user.dogAge}}`) resolve client-side at render time — UX must collect these in onboarding (§3) and the onboarding flow doubles as the Pyodide warmup mask. The fields in *my* `BaseStep` that aren't in arch's (`phase`, `estSeconds`, `concept`) are flagged to arch as additions; until then, treat them as UX-derived metadata that the lesson-shape rules in §2 and the daily-goal math in §5 depend on.

### 1.2 Layout — the persistent IDE (Codecademy three-pane)

Every step renders inside a **three-pane layout**, not eight different layouts. This is the single biggest copy from Codecademy and it's load-bearing for the "I'm actually coding" feel.

```
┌─────────────┬──────────────────────────┬────────────────────────────┐
│  SIDEBAR    │   PROMPT PANEL           │   IDE                      │
│  (chapters) │   (the step's content:   │   ┌──────────────────────┐ │
│             │    text, MC, blanks,     │   │ main.py    pets.txt  │ │
│  ▾ Loops    │    or "What does this    │   ├──────────────────────┤ │
│   ✓ L1      │    print?")              │   │ 1  pets = ["luna",   │ │
│   ✓ L2      │                          │   │ 2           "moose"] │ │
│   ● L3      │                          │   │ 3  for pet in pets:  │ │
│   ○ L4      │                          │   │ 4      print(pet)    │ │
│ ─────────── │   [ Hint ]   [ Skip ]    │   │                      │ │
│ 🔥 7 · 82xp │                  [→]     │   ├──────────────────────┤ │
│ ☀ 4/10 min │                          │   │ ▶ Run        ⌘↵      │ │
│             │                          │   ├──────────────────────┤ │
│             │                          │   │ OUTPUT               │ │
│             │                          │   │ luna                 │ │
│             │                          │   │ moose                │ │
│             │                          │   └──────────────────────┘ │
└─────────────┴──────────────────────────┴────────────────────────────┘
   200px            ~480px                          flex (≥ 480px)
```

**The IDE is always visible.** Even on `read` and `mc` steps where the user can't *change* anything, the IDE is populated, syntax-highlighted, and **runnable**. This is the move. Every step is a coding step, even when it's not.

**IDE chrome (consistent across all steps):**
- **File tabs** (top) — usually `main.py`, sometimes a read-only data file (`pets.txt`, `users.json`). Tabs don't disappear; they show what's loaded.
- **Editor** (middle) — CodeMirror 6 + Python mode. Line numbers, soft-wrap off, font: JetBrains Mono 14px. Read-only steps lock typing but keep selection + copy + scrolling. A small `🔒` icon on the gutter marks read-only.
- **Run bar** — `▶ Run` button, keyboard shortcut `⌘↵` / `Ctrl+↵`, and a small spinner during Pyodide execution. Below it: `Reset to starter` (subtle), and on `write`/`fix` steps a `Submit` button that runs hidden tests.
- **Output panel** — combined stdout + stderr, scrollable. A `Clear` button. When the step grader runs, a green `✓ Passed` or red `✕ Try again` chip appears at the top of the output panel — never as a modal.

**Per-step IDE state:**

| Step type    | IDE editor               | Run button         | Tabs                        |
| ------------ | ------------------------ | ------------------ | --------------------------- |
| `read`       | populated, **read-only** | enabled            | `main.py` (locked)          |
| `mc`         | populated if step has code, **read-only**; otherwise empty pane with a "no code yet" placeholder that fades back when next step has code | enabled if code exists | `main.py` |
| `fill`       | populated with `___` blanks rendered as inline editable chips inside the editor; rest of file read-only | disabled until all blanks filled | `main.py` |
| `predict`    | populated, **read-only**; user types prediction in prompt panel, then `Run` reveals actual output below their guess | enabled | `main.py` |
| `fix`        | populated with broken code, **fully editable**; bug-line gutter markers if `bugLines` set | enabled            | `main.py`                   |
| `write`      | populated with `starter`, fully editable; `Reset to starter` always visible | enabled (visible tests); `Submit` runs hidden tests | `main.py` + occasional data tabs |
| `reorder`    | IDE shows the *target shape* — empty editor + drop-zone hints; user drags blocks from prompt panel into editor lines | enabled once order is complete | `main.py` |
| `checkpoint` | same as `write`, with starter scaffolding for a slightly larger task (5–15 lines) | enabled + `Submit` | `main.py` |

**Two non-obvious rules:**

1. **The editor never empties between steps.** When the user advances from a `mc` to a `fill`, the IDE animates the relevant code into place rather than clearing. This sells the *"this is one continuous program I'm building"* fiction even though each step has its own code. Continuity is the feeling Codecademy nails.

2. **Run is always available, even when "wrong."** A user on a `fill` step who hasn't filled all blanks yet can still hit Run — the editor runs whatever's in it (probably crashes). This is *educational* not punishing: they see the `SyntaxError`, learn why blanks matter, and move on. Gating Run breaks the IDE feel.

**Mobile (<768px):** the three panes become a single-column stack with **the IDE as a sticky bottom drawer** (60% of viewport, swipe down to peek prompt, swipe up for full IDE). The Run button floats top-right of the drawer. Output is a scrollable region below the editor; the drawer never auto-collapses during execution.

### 1.3 The eight step types

> All eight step types render inside the **three-pane shell defined in §1.2** — sidebar (chapters) + prompt panel (left, ~480px) + IDE (right, flex). The schemas below describe content; the layout is constant. Where a sketch is shown, it shows only the prompt+IDE columns for compactness.

#### 1) `read` — lower the cognitive load before any production

```json
{ "kind": "read", "phase": "warmup",
  "prompt": "A `for` loop walks through every item in a list, one at a time.",
  "body": "```py\nfor name in pets:\n    print(name)\n```\nEach trip through, `name` becomes the next pet.",
  "continueLabel": "Got it", "estSeconds": 25, "xp": 5 }
```

```
┌──────────────────────┬──────────────────────────────────┐
│ Step 1/14   +5 XP    │ main.py  🔒                     │
│ ████░░░░░░░░░░░░░    ├──────────────────────────────────┤
├──────────────────────┤ 1  pets = ["luna", "moose"]      │
│ A `for` loop walks   │ 2  for name in pets:             │
│ through every item   │ 3      print(name)               │
│ in a list, one at    │                                  │
│ a time.              ├──────────────────────────────────┤
│                      │ ▶ Run  ⌘↵                        │
│ Try clicking Run →   ├──────────────────────────────────┤
│                      │ OUTPUT                           │
│      [ Got it → ]    │ (press Run)                      │
└──────────────────────┴──────────────────────────────────┘
```

When: warmup, position 0–2. The user can press Run on the read-only code to see output before clicking Continue — this is the "felt the loop" moment of the very first step. Correct = pressing Continue; there is no wrong. Target **20–35s**.

#### 2) `multiple-choice` — recall + prediction

```json
{ "kind": "multiple-choice", "phase": "warmup",
  "prompt": "Which of these is a list?",
  "choices": [
    { "id": "a", "label": "`{\"cat\", \"dog\"}`" },
    { "id": "b", "label": "`[\"cat\", \"dog\"]`" },
    { "id": "c", "label": "`(\"cat\", \"dog\")`" },
    { "id": "d", "label": "`\"cat, dog\"`" }
  ],
  "correctChoiceId": "b",
  "feedback": {
    "a": "Close — that's a `set`, no order, no duplicates.",
    "b": "Yes. Square brackets make a list.",
    "c": "That's a tuple — like a list, but you can't change it.",
    "d": "That's just one string with a comma in it." },
  "estSeconds": 30, "xp": 8 }
```

When: warmup or check. Wrong = choice-specific feedback (Codecademy rule), never just "wrong." Target **20–40s**.

#### 3) `fill-blank` — commit to a token without the blank page

```json
{ "kind": "fill-blank", "phase": "build",
  "prompt": "Complete the loop so it prints every pet.",
  "template": "for ___ in pets:\n    print(name)",
  "blanks": [{ "id": "b1", "answer": "name", "alternates": [], "caseSensitive": true }],
  "estSeconds": 45, "xp": 8 }
```

```
┌──────────────────────┬──────────────────────────────────┐
│ Step 5/14   +8 XP    │ main.py                          │
├──────────────────────┤ 1  pets = ["luna", "moose"]      │
│ Complete the loop    │ 2  for ┌──────┐ in pets:         │
│ so it prints every   │ 3      │ name │                  │
│ pet.                 │ 4      └──────┘                  │
│                      │ 5      print(name)               │
│ 💡 Hint              │   (other lines read-only)        │
│                      ├──────────────────────────────────┤
│      [ Check ⏎ ]     │ ▶ Run  ⌘↵                        │
│                      ├──────────────────────────────────┤
│                      │ OUTPUT  ✓                        │
│                      │ luna  / moose                    │
└──────────────────────┴──────────────────────────────────┘
```

When: build, position 3–8. The blanks are *inline editable chips inside the IDE itself* — the user is typing into the actual file, not a separate form field. Run is enabled even before all blanks are filled (it'll error — that's fine, it's instructive). Check uses the grader. Correct = exact (or alternate) match after `.trim()`; wrong surfaces case/alt-specific nudges. Target **30–60s**.

#### 4) `predict-output` — Khan's read-before-write rule

```json
{ "kind": "predict-output", "phase": "warmup",
  "prompt": "What does this print?",
  "code": "pets = [\"cat\", \"dog\"]\nfor p in pets:\n    print(p.upper())",
  "expected": "CAT\nDOG",
  "mode": "free-text",
  "estSeconds": 60, "xp": 8 }
```

When: warmup or build. Free-text matches normalized whitespace, case-insensitive line-by-line; `mode: "multiple-choice"` falls back to the MC schema. "I'm not sure" is an *honest* answer — no XP, no wrong-buzz — and drops the user into `read` mode for the same code. Target **40–75s**.

#### 5) `fix-bug` — the highest-leverage step type for AI-first builders

The one. AI-first builders rarely write code from a blank line — they read AI output and judge it. `fix-bug` trains exactly that.

```json
{ "kind": "fix-bug", "phase": "build",
  "prompt": "This crashes. Find the one bad line.",
  "code": "pets = [\"cat\", \"dog\", \"snail\"]\nfor pet in pets\n    print(pet)",
  "bugLines": [2],
  "expectedFixedOutput": "cat\ndog\nsnail",
  "hint": "Python uses one specific punctuation mark to start a block.",
  "estSeconds": 75, "xp": 12 }
```

```
┌──────────────────────┬──────────────────────────────────┐
│ Step 8/14   +12 XP   │ main.py                          │
├──────────────────────┤ 1  pets = ["cat","dog","snail"]  │
│ This crashes. Find   │ 2  for pet in pets    ⚠ click   │
│ the one bad line.    │ 3      print(pet)                │
│                      │   (fully editable)               │
│ 💡 Python uses one   ├──────────────────────────────────┤
│   specific punct     │ ▶ Run  ⌘↵                        │
│   mark to start      ├──────────────────────────────────┤
│   a block.           │ OUTPUT  ✕                        │
│                      │ SyntaxError: expected ':'        │
│      [ Submit ]      │ (line 2)                         │
└──────────────────────┴──────────────────────────────────┘
```

When: build, 2–3 per lesson minimum. **The IDE is the workbench** — the user edits broken code in place, hits Run, reads the actual Python error, and fixes it. The error message in the output panel is part of the lesson, not a popup. Correct = code runs AND output matches `expectedFixedOutput`. User can also flag the bad line in the gutter for a hint without editing. Target **60–90s**.

#### 6) `write-code` — the boot.dev exercise, but smaller

```json
{ "kind": "write-code", "phase": "build",
  "prompt": "Write a loop that prints each pet's name in uppercase.",
  "starter": "pets = [\"cat\", \"dog\"]\n# your code here",
  "tests": [{ "kind": "stdout-equals", "expected": "CAT\nDOG" }],
  "hiddenTests": [{ "kind": "with-input", "globals": { "pets": ["fish"] }, "expected": "FISH" }],
  "estSeconds": 90, "xp": 20 }
```

When: 1–2 per lesson max, always **after** a `fix-bug` and `fill-blank` on the same construct, so the user has typed the shape twice before producing it. Run = visible tests; Submit = visible + hidden. Target **75–120s**.

#### 7) `code-reorder` — drag-the-blocks (Brilliant + Parsons-problem)

```json
{ "kind": "code-reorder", "phase": "build",
  "prompt": "Drag the lines into the right order so it prints each pet.",
  "blocks": [
    { "id": "a", "code": "pets = [\"cat\", \"dog\"]" },
    { "id": "b", "code": "for pet in pets:" },
    { "id": "c", "code": "    print(pet)" }
  ],
  "distractors": [{ "id": "d", "code": "for pet of pets:" }],
  "correctOrder": ["a", "b", "c"],
  "estSeconds": 45, "xp": 8 }
```

When: build, great for indentation-heavy concepts. The distractor (`for pet of pets:` from JS muscle memory) is a teaching moment when picked. Target **40–75s**.

#### 8) `checkpoint-quiz` — the lesson outro, Duolingo "stats screen" feeder

```json
{ "kind": "checkpoint-quiz", "phase": "check",
  "prompt": "Quick check before you move on.",
  "questions": [
    { "qid": "q1", "kind": "multiple-choice", "prompt": "...", "choices": ["..."], "correctChoiceId": "b" },
    { "qid": "q2", "kind": "fill-blank", "template": "...", "blanks": ["..."] }
  ],
  "passThreshold": 0.66, "estSeconds": 90, "xp": 25 }
```

When: always last step. 2–3 short checks → one screen, feeds the end-of-lesson stats card. Failing retries only the missed sub-questions, not the whole lesson (hard rule, ADHD non-punishment). Target **60–120s**.

## 2. Lesson shape — how steps compose

A **lesson** is 12–20 steps, ~5–8 minutes total, sequenced in three phases:

```
Phase           Length  Step mix                                 Why
───────────────────────────────────────────────────────────────────────────────
WARMUP   (≈25%) 3–5     read • mc • predict-output               Lower entry friction. Always start with a win.
BUILD    (≈60%) 7–12    fill-blank • code-reorder • fix-bug      Codecademy "minimum-blank-page" path.
                        • write-code (1–2 max)
CHECK    (≈15%) 1–3     mc (recall) • checkpoint-quiz            Confirm + reward + roll into stats card.
```

**Canonical opening sequence (rule of three):**
`read → mc → read → predict-output → fill-blank → fix-bug → code-reorder → fix-bug → write-code → mc → checkpoint-quiz`

Why this works:
- **Two reads before production** (Khan): the construct is in working memory before they type.
- **`fill-blank` → `fix-bug` → `write-code`**: escalating commitment, no leaps.
- **`fix-bug` between production steps**: trains the AI-first muscle (read → judge → repair).
- **`write-code` rare, always after the same shape has been seen three times**: kills the "no idea where to start" cliff.

Lesson target: 5–8 min (Duolingo budget), median 6. A chapter = 4–7 lessons (~30–45 min), same total as today but split into resumable units.

## 3. Onboarding flow — five screens, one schema saved

```
[1] WELCOME → [2] GOAL → [3] LEVEL CHECK → [4] PERSONALIZATION → [5] DAILY GOAL → /learn
```

1. **Welcome** — H1 + single button: *"You're going to learn Python. With AI as your co-pilot, not your crutch."* ~15s.
2. **Goal with AI** — One MC, four options (no "Other"): side project / team tools / startup / curious. Drives example tone; saved as `user.goal`.
3. **Level check** — Three MCs framed "so we don't bore you," not "to test you." → `user.level`.
4. **Personalization (skippable)** — Pet name, team/company, hometown. Substitutes into example data (`pets = ["luna", "moose"]` not `["cat", "dog"]`). Biggest single retention lever in onboarding.
5. **Daily goal** — Four cards (5/10/20/40 min), Regular preselected. "Change anytime."

**Schema saved (additions to `lib/types.ts`):**

```ts
type UserProfile = {
  name: string;
  goal: "side-project" | "team-tools" | "startup" | "curious";
  level: "absolute-beginner" | "some-exposure" | "rusty";
  flavor?: { pet?: string; team?: string; city?: string };
  dailyGoalMinutes: 5 | 10 | 20 | 40;
  createdAt: string;
  reducedMotion: boolean;
  soundEnabled: boolean;
};
```

Screens 2–4 skippable (small link, never bold). Screen 5 has defaults pre-selected.

## 4. Progression UI — the **Codecademy sidebar**, not the path

Options on the table: Duolingo path, Codecademy sidebar, GitHub-graph DAG. **Pick: Codecademy sidebar.**

For ADHD + AI-first builders:
- **Duolingo path** optimizes for surprise + linear narrative. Our user wants to *jump back to chapter 4* while doing chapter 7 — the path fights that.
- **DAG** looks beautiful in screenshots and scares actual learners. Choice paralysis kills ADHD users; a visible "next" is sacred.
- **Sidebar** gives the same "where am I" cue with zero scroll tax, supports keyboard nav (`j/k`), and degrades to a top-bar on mobile.

```
┌──────────────┬───────────────────────────────────┐
│ CHAPTERS     │  Lesson 3 of 5 · Loops            │
│              │  Step 6 of 14                     │
│ ▸ 1 Hello    │                                   │
│ ▸ 2 Variables│  ████████████░░░░░░    +8 XP     │
│ ▾ 3 Loops    │                                   │
│   ✓ L1 Setup │   [ MAIN STEP CONTENT HERE ]      │
│   ✓ L2 ForLp │                                   │
│   ● L3 Range │                                   │
│   ○ L4 While │                                   │
│   ○ L5 Quiz  │                                   │
│ ▸ 4 Lists    │                                   │
│ ─────────────│                                   │
│ 🔥 7  •  82xp│   [ Hint ]  [ Brain dump ]        │
│ ☀ 4 / 10 min │   [ Skip ]                  [→]   │
└──────────────┴───────────────────────────────────┘
```

- Filled circle = in progress, check = done, open = locked-but-visible. Locked items *clickable to preview*, never hidden — ADHD users need the whole shape.
- Bottom of sidebar: streak count, today's XP / daily goal as one small dial.
- Mobile (<768px): sidebar collapses to a bottom sheet, default-collapsed; current step takes full screen.

## 5. Daily goal mechanic (built FOR ADHD, not against it)

1. **Goal is a floor, not a target.** UI says "today: 4 / 10 min." Passing 10 turns the dial green and adds a tiny `+`. No "you've done your daily, go away."
2. **Missing it never breaks streak.** Streaks count *days touched*, not days completed. Open a brain-dump note → streak counted. Kills the guilt-spiral.
3. **One-click resume.** Persistent "Continue: Chapter 3 · Step 6" pill in top nav and on `/learn`. Two-next-moves rule: pill on `/learn` always shows `[ Continue ]` AND `[ Pick something else → ]`.
4. **Visible session timer, invisible deadlines.** Top-right shows elapsed minutes for the session. Never counts *down*; green threshold appears when crossed, like a slow sunrise.
5. **Exceeded → ember bank, not nag.** Going over banks "embers" (max 2). A missed day burns one ember instead of breaking streak. Uses the existing `embers` field in `StreakState`.

Copy:
- On goal-met: `Nice. Today's done. You're 2 min in.` (not `Goal complete!!!`)
- On streak-saved-by-ember: `Yesterday's ember kept the fire going.`
- On streak-broken: `Streak reset to 1. New start, same you.` (welcome-back, never guilt-back)

## 6. Sound + animation design

| Trigger                         | Sound                       | Animation                              | Length    | Opt-out                  |
| ------------------------------- | --------------------------- | -------------------------------------- | --------- | ------------------------ |
| Step correct                    | Soft 800Hz pluck            | Check-mark draw-on, ease-out           | 140ms     | settings + 1st miss      |
| Step wrong (small)              | Muted 240Hz thunk           | Input shakes 6px once                  | 200ms     | settings + reduced-motion|
| Step wrong (multiple in a row)  | Silent (no piling on)       | Hint expand from 💡 button             | 220ms     | always silent            |
| Lesson complete                 | 3-note major-third arpeggio | Confetti burst (12 particles)          | 600ms     | reduced-motion → fade    |
| Streak +1                       | Whoosh + ember pop          | 🔥 number tick-up + glow pulse          | 400ms     | reduced-motion → tick    |
| Daily goal crossed              | Single low chime            | Dial fills green                       | 700ms     | settings                 |
| XP earned (every step)          | Tiny tick (–6dB)            | XP bar slides + number floats up       | 250ms     | settings                 |
| Brain dump opened               | None                        | Slide-in from right                    | 180ms     | reduced-motion → instant |

Rules:
- Sounds default **off** for first session; prompted at end of lesson 1: *"Want sounds? Some people find them satisfying."* No dark pattern.
- Reduced motion follows `prefers-reduced-motion` AND a settings toggle. When on, replace movement with opacity fades — never remove visual feedback entirely (deaf + motion-sensitive users still need it).
- All sounds ≤ 600ms, ≤ 70dB at default volume, compressed `.webm`/`.mp3` pairs (~2KB each).
- **No sound or animation gates progress.** Muted + reduced-motion gets the same XP and visual state, just instant.

## 7. Information architecture

Hierarchy: `User → Profile → Course → Chapter → Lesson → Step → (Hint, Solution, Attempt → BrainDump)`. `Lesson` and `Step` are new (previously `Exercise`). Static content lives under `/lib/generated`; attempts and progress live in localStorage with write-through to server.

**Where things live:**

| Data                              | localStorage | Server | Roll-up                       |
| --------------------------------- | :----------: | :----: | ----------------------------- |
| Step drafts (in-progress code)    | ✓            | —      | n/a                           |
| Step attempts (correct/wrong)     | ✓            | ✓      | → lessonStatus, → totals.xp   |
| Brain-dump notes                  | ✓            | sync¹  | none                          |
| Streak state                      | ✓            | ✓      | server is truth               |
| User profile                      | ✓            | ✓      | server is truth               |
| Per-session timer                 | sessionStorage only | — | n/a                       |
| Last-visited pointer              | ✓            | ✓      | drives Resume pill            |

¹ Brain-dump syncs *opportunistically*, never blocks. If offline, it stays local.

**Completion semantics:**
- `Step` is `passed` after one correct attempt.
- `Lesson` is `complete` when ≥ 90% of its non-`read` steps pass AND its checkpoint-quiz passes (90% lets users skip one stuck step without lockout).
- `Chapter` is `complete` when all lessons are complete.
- `conceptsTouched` = union of `concept` across all attempted steps.

## 8. Accessibility + reduced-motion baseline

- [ ] Every interactive element reachable by Tab in source order; 2px focus ring at 3:1 contrast.
- [ ] All step types keyboard-operable:
  - MC: `1`–`9` selects, `Enter` checks.
  - fill-blank: `Tab` between blanks, `Enter` checks.
  - code-reorder: `Space` to grab, arrows to move, `Enter` to drop. Drag is *additive*.
  - fix-bug: arrows nav gutter, `Enter` flags "this line is bad."
- [ ] Color is never the only signal. Correct/wrong always has icon (✓/✕) + text label.
- [ ] Contrast ≥ 4.5:1 body, ≥ 3:1 large + UI controls. Verified light + dark.
- [ ] `prefers-reduced-motion: reduce`: no shake, no confetti, no slide-ins → 100ms opacity fades.
- [ ] `<pre><code lang="python">` for screen readers; output pane has `aria-live="polite"`.
- [ ] All form inputs have visible labels.
- [ ] Feedback: `aria-live="assertive"` blocking, `polite` non-blocking.
- [ ] One `h1` per page; step prompts are `h2`.
- [ ] Idle hints (3/5-min) pausable via Settings → "I work slowly, give me space."
- [ ] Streak break is **never** announced via `assertive` — hostile to users opening the app after a hard week.

## 10. Three redlines (UX patterns we will not adopt)

1. **No hearts / no lives.** Duolingo's heart loss converts well for kids learning Spanish; it converts *out* of an ADHD adult learning a hard skill. Ember banking instead — addition, not subtraction.

2. **No leaderboards by default.** Leagues are extrinsic motivation that crowds out intrinsic. AI-first builders mostly build alone; comparing to faster strangers is anti-flow. (Optional opt-in cohorts later, never default.)

3. **No paywall in chapters 1–5, no "free limit, come back tomorrow" gate.** Mid-flow paywall is the biggest churn point in the category. Free tier = 5 chapters end-to-end, full features. Paywall starts at ch6, surfaced once, never again per session. Goal-met / streak-met cards never link to upgrade — they protect the win moment.

## TL;DR

- The product's atom is the **Step**, not the Exercise — eight types (`read`, `mc`, `fill`, `predict`, `fix`, `write`, `reorder`, `checkpoint`), each 30–90s, normalized to `StepAttempt`. Type names + grader/hint shapes are reconciled with `arch` (03-architecture.md §1).
- **Every step renders inside a persistent three-pane shell** — sidebar + prompt panel + Codecademy-style IDE on the right. The IDE is *always populated and runnable*, even on `read` and `mc` steps. No tool-switch between "reading about code" and "writing code" — they're the same screen.
- A **lesson is 12–20 steps in three phases** (warmup ~25%, build ~60%, check ~15%); canonical sequence is *two reads + one MC* before any production step. `fix` is the highest-leverage type for AI-first builders (read-and-judge muscle), used 2–3× per lesson — more than `write`.
- Progression UI is the **Codecademy sidebar** (not Duolingo path / not DAG); daily goal is a *floor*, streaks count days-touched, missing it never punishes. Embers absorb missed days.
- Three redlines: **no hearts, no leaderboards by default, no mid-flow paywall in chapters 1–5** — every reward is additive, every nudge is welcome-back not guilt-back.
